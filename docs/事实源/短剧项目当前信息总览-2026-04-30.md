### 2.1 收入模型

当前收入先按 `平台分成 + 海外广告分成 + 保底` 三类模型并行记录。所有估算只作为内部决策口径，最终以平台后台、商务通知或签署协议为准。

#### 2.1.1 通用 IAA 估算

```text
单集收入 = 播放量 / 1000 * 创作者侧有效 RPM
```

用途：用于没有明确有效时长和平台分账单价时的粗估，不替代红果、火龙、YouTube 的实际结算公式。

#### 2.1.2 红果漫剧免费分成

来源：https://bytedance.larkoffice.com/docx/GrVcdz5hDoFw1WxpUXgcJ9VCnMh?vwb=1.0.0.5016&is_whiteboard_x=true&debug=1&whiteboardDebugMode=1

```text
单部漫剧免费分成 = 当月新增有效时长 * 时长单价 * 漫剧类型系数 * 版权系数
```

| 漫剧类型 | 系数 |
|---|---:|
| 仿真人剧 | `60` |
| 3D 动画漫剧 | `50` |
| 2D 动画漫剧 | `40` |
| 表情包动态漫剧 | `10` |
| 动态解说漫剧 | `5` |
| 静态解说漫剧 | `1` |

| 版权类型 | 系数 |
|---|---:|
| 全版权（特定权利 + 付费免费一体合作） | `2` |
| 非全版权（除上述之外的其他合作模式） | `1` |

项目判断：

- 红果分成强依赖 `有效时长` 和 `时长单价`，回传表必须补这两个字段；
- 同等播放下，仿真人剧、3D 动画漫剧、2D 动画漫剧优先级高于解说类；
- 全版权系数更高，但会影响海外分发和后续多平台策略，签前必须做冲突检查。

#### 2.1.4 YouTube 创作激励 / 广告分成

来源：YouTube Help 官方文档，包括 `YouTube Partner Program overview & eligibility`、`YouTube partner earnings overview`、`YouTube Shorts monetization policies`。YouTube 没有固定统一 RPM，收益受国家 / 地区、广告填充、观看结构、音乐版权、内容合规和频道是否加入 YPP 影响。

准入门槛：

```text
YPP 常规门槛 = 1000 订阅 + 过去 12 个月 4000 有效公开视频观看小时
YPP Shorts 门槛 = 1000 订阅 + 过去 90 天 1000 万有效公开视频 Shorts 观看
```

| 类型 | 收入模型 | 创作者分成系数 | 当前用途 |
|---|---|---:|---|
| Watch Page Ads | 创作者收入 = Watch Page 可分配广告净收入 * `55%` | `0.55` | 适合短剧合集、长切片和系列页 |
| Shorts Feed Ads | 创作者收入 = Shorts Creator Pool 分配收入 * `45%` | `0.45` | 适合海外渠道最小闭环、快速测题材和包装 |
| YouTube Premium Shorts | 创作者收入 = 分配给 Shorts 创作者的 Premium 净收入 * `45%` | `0.45` | 按各国订阅 Shorts 观看占比分配 |
| Commerce / 粉丝付费 | 创作者收入 = 净收入 * `70%` | `0.70` | 账号成熟后的补充收入 |

项目判断：

- 未进入 YPP 前，YouTube 播放量只作为渠道反馈，不能直接算收入；
- Shorts 更适合先测题材、标题、封面和节奏；高表现素材再改成长视频合集或系列页；
- 数据表必须记录 `视频形态 / 国家地区 / 是否原创 / 是否用音乐 / 是否入 YPP / 估算 RPM / 实际结算`。

当前执行口径（2026-04-30 更新）：

- 日间工作优先使用 `libtv / lib 接口`，原因是并发高、速度快，适合作为主生产通道；
- 真人剧相关工作同样优先使用 `libtv / lib 接口`，减少通道切换和等待时间；
- 晚间挂机任务使用 `即梦 + lovart` 两端作为补充通道，适合低人工值守的长队列任务；
- 火山引擎 API 保留为赶进度、稳定批量或主通道异常时的备用选项。

### 2.5 发布优先级

- `P0`：海外分发最小闭环，创建 YouTube、TikTok、X（Twitter / 推特）等渠道账号，进行素材投放，补齐发布、反馈、收益和成本数据；
- `P1`：在海外渠道出现可解释反馈后，评估红果 / 抖音集团免费生态的非独家分成合作；
- `P2`：抖音原生、火龙漫剧 / 腾讯视频作为二阶段扩量或平台对照线，逐项确认首发、独家、授权期限和分成公式；
- `P2+`：若腾讯签约顺利，争取至少签下 `1` 个 `A 级保底`，但不得影响非排他和海外渠道验证空间。

#### 2.6.2 火龙漫剧 / 腾讯视频政策

基础分账公式：

```text
分成收益 = 有效播放时长 * 时长单价 * 类型系数 * 版权系数
```

| 平台 | 内容类型系数 | 版权系数 | 当前项目判断 |
|---|---|---|---|
| 火龙漫剧 | AI 仿真人剧 / 3D / 2D：`1`；解说漫：`0.5` | 独家：`2`；非独家首发：`1` | 优先做非解说型动态交互内容；解说漫收益系数低，不应作为主生产形态 |
| 腾讯视频 | AI 仿真人剧 / 3D / 2D：`1`；解说漫：`0.3` | 原政策表未单独列清腾讯侧版权系数，签约前需商务确认 | 若走腾讯生态，必须提前确认是否要首发、是否影响其他平台发布 |

定义与要求：

- `有效播放时长` 以火龙漫剧和腾讯视频平台最终核算为准，会扣除刷量、数据作假等无效时长；
- `独家` 指火龙漫剧和腾讯视频拥有该内容独占专有传播权；
- `非独家首发` 指内容未在全网其他平台上线，且在火龙漫剧和腾讯视频上传时间不得晚于全网其他平台；
- 内容建议时长为 `1~3 分钟`，横屏竖屏不限，AI 真人漫剧也可上传；
- 内容需符合国家政策法规及审查标准，版权归属清晰，专辑需上传全集或定期更新；
- 账号需完成实名认证，腾讯视频创作平台信用分为 `100`，已开通专辑原创声明；企业资质需企业认证。

精品独家保底政策：

| 评级 | 纯现金保底 | 现金 + 算力 | 关键限制 |
|---|---:|---:|---|
| S+ | 单分钟 `10000 元`，上不封顶 | 单分钟 `9000 元以上 + 2000 元算力` | 仅适用于 AI 仿真人剧、3D 动漫；上线前未在全网任何其他平台上线 |
| S | 单分钟 `3000 元以上` | 单分钟 `2800 元 + 450 元算力` | 第一年为火龙漫剧与腾讯视频独家，第二至第三年非独家，第四年至永久恢复独家 |
| A | 单分钟 `1000 元以上` | 单分钟 `800 元 + 450 元算力` | 地域范围为全球 |

其他激励：

- 火龙漫剧播放热榜激励：每周按单内容播放热度奖励，`top1-3` 为 `3 万元`，`top4-10` 为 `1.5 万元`，`top11-20` 为 `1 万元`；
- 火龙漫剧持续创作激励：单内容连续两个月更新内容时长超过 `60 分钟`，且单月累计播放时长超过 `10 万分钟`，可获得最高 `2 万元`；

## 3. 生产链路标准

来源：`docs/59-小说到分集脚本Prompt链路与StoryWeaver嵌入-v1.md`、`<StoryWeaver 项目>/docs/short-drama-production-sop.md`

### 3.1 标准 7 步

| 步骤 | 输出文件 | 过门标准 |
|---|---|---|
| Story DNA 提取 | `01-story-dna.md` | 一句话说清主冲突 |
| 季纲规划 | `02-season-arc.md` | 每 10 集有阶段反转 |
| 分集蓝图 | `03-episode-blueprint.md` | 每集都能独立留钩 |
| 分集脚本 | `04-script-pack.md` | 每 15~20 秒有可见变化 |
| 分镜拆解 | `05-storyboard-pack.md` | 单集镜头数可控 |
| StoryWeaver 入参 | `06-storyweaver-input.json` | 可直接进入执行引擎 |
| 质量门 | `07-quality-gate.md` | 不合格不扩集 |

### 3.2 单集节拍

90 秒模板：

- `0~3 秒`：开头钩子；
- `3~15 秒`：最少背景；
- `15~35 秒`：冲突升级；
- `35~60 秒`：首次反转；
- `60~80 秒`：情绪兑现；
- `80~90 秒`：集末卡点。

### 3.3 低返工规则

- 先压 `5~6` 句六句稿，再拆长稿；
- 每个关键镜头只保留 `1` 个主标签和 `1~2` 个次标签；
- 镜头重点错了，先修主标签权重，不先换模型；
- 信息太满，先砍次标签；
- 成本超支，先砍镜头数和群像难镜，不先升模型档位。

### 3.4 StoryWeaver AI 已有工作流功能与操作 SOP

定位：StoryWeaver AI 是当前短剧生产执行工具，覆盖 `项目 -> 分集 -> 剧本 -> 资产 -> 分镜 -> 图片 -> 视频 -> 音频 -> 剪映工程 / 素材包导出` 的生产链路。它不是选题工具，主要负责把已确认的 run 包变成可质检、可导出的生产工程。

#### 3.4.1 主工作流

#### 4.2.1 IP 授权获取逻辑链路

参考工具：`.codex/skills/shortdrama-ip-scraper/`、`wechat-article-exporter-master/`。

执行链路：

1. `IP 发现`：用 `shortdrama-ip-scraper` 从短剧创作者版权 IP 列表抓取全量候选，先形成 `ip_id / 作品名 / 分类 tag / 作者 / 评分 / 挑选次数 / 完结状态 / 字数 / 读者数 / raw_json` 基础表。
2. `本地筛选`：优先筛 `高评分 + 低挑选次数 + 字数适合短剧改编 + 分类贴合 cand-22 / cand-24 / cand-20` 的 IP，不直接按平台榜单热度开跑。
3. `权利字段补齐`：从 `raw_json` 和版权中心详情页补 `版权来源 / 授权入口 / 申请状态 / 可申请范围 / 是否已有挑选或授权记录`，仍缺字段则标为 `待人工确认`。
4. `外部证据留存`：用 `wechat-article-exporter-master` 导出相关公众号文章、公告、推荐文、授权说明或运营披露，留存为 HTML / Markdown / JSON 等可追溯证据。
5. `证据归档`：每个候选 IP 至少保留一条 `版权中心记录` 和一条 `外部证据记录`；证据只写相对路径，不写账号、cookie、token 或个人敏感信息。
6. `授权判断`：把候选 IP 写入授权状态表，给出 `可内部验证 / 可发布 / 可商用 / 需授权 / 不可碰` 五档结论。
7. `进入 run 包`：只有 `可内部验证` 及以上的 IP 可以进 P0 run 包；只有 `可发布 / 可商用` 的 IP 才能进入 P1 / P2 对外发布或平台对照。

判断逻辑：

| 判断项 | 通过条件 | 不通过处理 |
|---|---|---|
| 是否可内部验证 | 版权中心可见、来源可追溯，且没有明显禁止改编 / 禁止生成 / 排他限制 | 只进观察池，不进正式 run 包 |
| 是否可发布 | 有明确授权入口、申请记录或平台可发布范围说明 | 停在 P0，不进入 P1/P2 |
| 是否可商用 | 授权范围覆盖商业化、分发平台、改编形态和使用期限 | 不做收益承诺，不签外部合作 |
| 是否需授权 | IP 有潜力但授权范围未明、联系人未确认或外部证据不足 | 标记 `需授权`，由项目负责人推进 |
| 是否不可碰 | 存在排他、侵权争议、来源不明、平台限制或证据冲突 | 标记 `不可碰`，不得进入生产 |

最小入表字段：

| 字段 | 说明 |
|---|---|
| `ip_id` | 对应 IP 全量表 |
| `作品名` | IP 标题 |
| `候选方向` | cand-22 / cand-24 / cand-20 / 其他 |
| `版权中心证据` | 版权中心记录相对路径或记录 ID |
| `公众号证据` | wechat-article-exporter 导出的文章相对路径 |
| `授权状态` | 可内部验证 / 可发布 / 可商用 / 需授权 / 不可碰 |
| `可用范围` | P0 / P1 / P2 |

## 6. 当前下一步动作

### 6.1 目标调整

鉴于当前生产和数据现状，下一阶段目标先从“验证能否规模化赚钱”调整为更实际的 `盈亏打平`。所有新增动作都必须服务两件事：

1. 增加可结算收入来源；
2. 降低生产成本、废片率和人工摩擦，使 ROI 逐步大于 `1`。

### 6.2 最高优先级

| 任务 | 操作 | 输出 / 验收 | 判断口径 |
|---|---|---|---|
| 渠道拓展 | 创建海外分发渠道账号，如 YouTube、TikTok、X（Twitter / 推特）等，并进行素材投放 | 至少形成账号清单、发布素材清单、发布时间、链接、播放 / 互动 / 收益记录 | 先补全数据构建最小闭环，再考虑增效 |
| 渠道评估 | 统计当前分发渠道、发布节奏、发布数量及反馈情况 | 建立渠道评估表，按渠道记录播放、互动、粉丝增长、转化、收益、违规 / 限流情况 | 判断各渠道是否继续押注、低成本保留或放弃 |
| 成本效益分析 | 对比成片和半成片成本与收益 | 每个样本记录制作耗时、模型成本、人工成本、发布反馈和收益 | 若半成片收益与成片相差不大但耗时少，优先制作半成片 |
| 成本优化 | 降低废片率、更换模型、提高自动化程度 | 记录废片率、重跑次数、模型通道、自动化覆盖率和单集成本变化 | 目标是让 ROI 大于 `1` |
| 项目验证 | 下个月争取看到初步闭环并结算收益 | 至少拿到一轮渠道发布数据、收益数据和成本表 | 若腾讯签约顺利，争取至少签 `1` 个 `A 级保底` |
